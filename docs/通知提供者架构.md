# 通知提供者架构文档

## 概述

通知系统已经重构为独立的项目结构，类似于云厂商适配器的架构模式。每个通知渠道（如邮件、钉钉）都是独立的项目，便于维护、测试和扩展。

## 架构设计

### 项目结构

```
src/
├── FeeQuery.Notifications/              # 通知提供者根目录
│   ├── DingTalk/                        # 钉钉通知提供者
│   │   ├── FeeQuery.Notifications.DingTalk.csproj
│   │   └── DingTalkNotificationProvider.cs
│   └── Smtp/                            # SMTP 邮件通知提供者
│       ├── FeeQuery.Notifications.Smtp.csproj
│       └── SmtpNotificationProvider.cs
└── FeeQuery.Shared/
    └── Interfaces/
        └── INotificationProvider.cs     # 通知提供者接口
```

**设计理念**：
- 与云厂商适配器架构一致（`FeeQuery.Providers/Alibaba`, `Tencent` 等）
- 所有通知提供者集中在 `FeeQuery.Notifications` 目录下
- 每个通知渠道都是独立的子目录和项目
- 项目命名简洁：`FeeQuery.Notifications.{Provider}`

### 核心接口

所有通知提供者都实现 `INotificationProvider` 接口：

```csharp
public interface INotificationProvider
{
    /// <summary>
    /// 提供者类型标识（如 "email", "dingtalk"）
    /// </summary>
    string ProviderType { get; }

    /// <summary>
    /// 发送通知
    /// </summary>
    Task<bool> SendAsync(string title, string content, string configJson, CancellationToken cancellationToken = default);

    /// <summary>
    /// 测试连接配置
    /// </summary>
    Task<(bool Success, string? ErrorMessage)> TestConnectionAsync(string configJson, CancellationToken cancellationToken = default);
}
```

## 已实现的通知提供者

### 1. SmtpNotificationProvider（SMTP 邮件通知）

**项目**: `FeeQuery.Notifications.Smtp`
**位置**: `src/FeeQuery.Notifications/Smtp/`

**依赖**:
- FeeQuery.Shared
- Microsoft.Extensions.Logging.Abstractions

**配置格式** (JSON):
```json
{
  "SmtpServer": "smtp.example.com",
  "Port": 587,
  "Username": "your-email@example.com",
  "Password": "your-password",
  "From": "noreply@example.com",
  "To": "admin@example.com;user@example.com",
  "EnableSsl": true
}
```

**特性**:
- 支持 SMTP 协议
- 支持 SSL/TLS 加密
- 支持多收件人（用分号分隔）
- 详细的错误诊断

### 2. DingTalkNotificationProvider（钉钉通知）

**项目**: `FeeQuery.Notifications.DingTalk`
**位置**: `src/FeeQuery.Notifications/DingTalk/`

**依赖**:
- FeeQuery.Shared
- Microsoft.Extensions.Logging.Abstractions
- Microsoft.Extensions.Http

**配置格式** (JSON):
```json
{
  "WebhookUrl": "https://oapi.dingtalk.com/robot/send?access_token=xxx",
  "Secret": "SEC...",
  "AtAll": false
}
```

**特性**:
- 支持普通文本消息
- 支持交互式卡片（ActionCard）消息，带按钮操作
- 支持钉钉签名验证（Secret）
- 支持 @所有人功能

**扩展功能 - ActionCard**:

DingTalkNotificationProvider 提供了额外的 `SendActionCardAsync` 方法，用于发送带按钮的交互式卡片：

```csharp
public async Task<bool> SendActionCardAsync(
    string title,
    string content,        // Markdown 格式
    List<ActionCardButton> buttons,
    string configJson,
    CancellationToken cancellationToken = default)
```

按钮定义：
```csharp
public class ActionCardButton
{
    public string Title { get; set; } = string.Empty;
    public string ActionUrl { get; set; } = string.Empty;
}
```

## 使用方式

### 1. 在 Program.cs 中注册

```csharp
using FeeQuery.Notifications.Smtp;
using FeeQuery.Notifications.DingTalk;

// 注册通知提供者（作为单例）
builder.Services.AddSingleton<INotificationProvider, SmtpNotificationProvider>();
builder.Services.AddSingleton<INotificationProvider, DingTalkNotificationProvider>();
```

### 2. 在服务中使用

通过依赖注入获取所有通知提供者：

```csharp
public class MyService
{
    private readonly IEnumerable<INotificationProvider> _providers;

    public MyService(IEnumerable<INotificationProvider> providers)
    {
        _providers = providers;
    }

    public async Task SendNotification(string channelType, string title, string content)
    {
        // 根据类型查找对应的提供者
        var provider = _providers.FirstOrDefault(p => p.ProviderType == channelType);
        if (provider != null)
        {
            await provider.SendAsync(title, content, configJson);
        }
    }
}
```

### 3. 使用 NotificationService（推荐）

`NotificationService` 已经封装了提供者的查找和调用逻辑：

```csharp
public class BalanceAlertService
{
    private readonly NotificationService _notificationService;

    public async Task SendAlert()
    {
        // 自动根据通道类型查找提供者并发送
        await _notificationService.SendNotificationAsync(
            channelType: "email",
            title: "预警标题",
            content: "预警内容"
        );
    }
}
```

## 添加新的通知提供者

### 步骤 1: 创建新项目

```bash
cd src/FeeQuery.Notifications
mkdir {ProviderName}
cd {ProviderName}
dotnet new classlib -n FeeQuery.Notifications.{ProviderName} -f net10.0
cd ../../..
dotnet sln add src/FeeQuery.Notifications/{ProviderName}/FeeQuery.Notifications.{ProviderName}.csproj
```

### 步骤 2: 添加依赖

```bash
cd src/FeeQuery.Notifications/{ProviderName}
dotnet add reference ../../FeeQuery.Shared/FeeQuery.Shared.csproj
dotnet add package Microsoft.Extensions.Logging.Abstractions
# 根据需要添加其他包
```

```csharp

    {
        _logger = logger;
    }

    public async Task<bool> SendAsync(string title, string content, string configJson, CancellationToken cancellationToken = default)
    {
        // 实现发送逻辑
    }

    public async Task<(bool Success, string? ErrorMessage)> TestConnectionAsync(string configJson, CancellationToken cancellationToken = default)
    {
        // 实现测试逻辑
    }

```

### 步骤 4: 在 Web 项目中注册

```bash
cd ../../FeeQuery.Web
dotnet add reference ../FeeQuery.Notifications/{ProviderName}/FeeQuery.Notifications.{ProviderName}.csproj
```

在 Program.cs 中添加：


## 架构优势

### 1. 解耦合
- 通知提供者与核心业务逻辑完全分离
- 每个提供者都是独立的项目，互不影响
- 符合依赖倒置原则（DIP）

### 2. 易于测试
- 可以独立测试每个通知提供者
- 可以轻松 Mock INotificationProvider 进行单元测试
- 无需启动整个应用即可测试单个提供者

### 3. 易于维护
- 添加或修改通知渠道不影响核心代码
- 每个提供者有自己的依赖项，不会污染主项目
- 清晰的项目结构便于定位问题

### 4. 可扩展性
- 遵循开闭原则（OCP）：对扩展开放，对修改关闭
- 添加新通知渠道只需添加新项目和注册
- 支持第三方提供通知提供者实现

### 5. 灵活性
- 可以为特定提供者添加扩展功能（如 DingTalk 的 ActionCard）
- 支持运行时动态加载通知提供者
- 可以根据环境（开发/生产）注册不同的提供者

## 与云厂商架构的一致性

通知提供者架构完全参考了云厂商适配器的设计模式：

| 特性 | 云厂商适配器 | 通知提供者 |
|------|-------------|-----------|
| 接口 | `ICloudProvider` | `INotificationProvider` |
| 根目录 | `FeeQuery.Providers/` | `FeeQuery.Notifications/` |
| 项目命名 | `FeeQuery.Providers.{Vendor}` | `FeeQuery.Notifications.{Provider}` |
| 目录结构 | `src/FeeQuery.Providers/{Vendor}/` | `src/FeeQuery.Notifications/{Provider}/` |
| 注册方式 | 单例注册 | 单例注册 |
| 依赖管理 | 独立项目依赖 | 独立项目依赖 |
| 工厂模式 | `CloudProviderFactory` | `NotificationService` |

**对比示例**：
```
云厂商:
src/FeeQuery.Providers/
├── Alibaba/FeeQuery.Providers.Alibaba.csproj
├── Tencent/FeeQuery.Providers.Tencent.csproj
└── Volcano/FeeQuery.Providers.Volcano.csproj

通知提供者:
src/FeeQuery.Notifications/
├── DingTalk/FeeQuery.Notifications.DingTalk.csproj
├── Smtp/FeeQuery.Notifications.Smtp.csproj
└── WeChat/FeeQuery.Notifications.WeChat.csproj  (示例)
```

这种一致性使得开发者可以快速理解和扩展通知系统，无需学习新的架构模式。

## 故障排除

### 问题：找不到通知提供者

**原因**: 提供者未在 Program.cs 中注册

**解决方案**:
```csharp
builder.Services.AddSingleton<INotificationProvider, EmailNotificationProvider>();
```

### 问题：ActionCard 按钮点击无响应

**原因**: 未配置正确的 BaseUrl 或未启用 Controllers

**解决方案**:
1. 在 `appsettings.json` 中配置 `FeeQuery:BaseUrl`
2. 确保 Program.cs 中调用了 `builder.Services.AddControllers()` 和 `app.MapControllers()`

### 问题：邮件发送失败

**常见原因**:
- SMTP 服务器地址或端口错误
- 用户名或密码错误
- 未启用 SSL（某些服务器要求）
- 防火墙阻止 SMTP 端口

**调试方法**:
使用 `TestConnectionAsync` 方法测试配置，查看详细错误消息。
